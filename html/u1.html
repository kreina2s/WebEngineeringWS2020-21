1.1 Fachliche Argumentation über Erfolgsprinzipien des WWW

Mit welchen fachlichen Argumenten wurde das WWW-Proposal von TBL abgelehnt?

Zunächst hat TBLs Vorgesetzter die Wagheit seines Konzeptes nicht verstanden, da er es mit der Evolvierbarkeit verwechselt hat. TBL strebte eine Evolvierbarkeit unterschiedlicher Technologien unabhängig voneinander an (HTTP, HTML, URI). Diese Weiterentwicklung sollte ohne zentrale Kontrolle erfolgen.

Laut eines ehemaligen Mitgliedes des Programmkomitees der weltgrößten Hypertext-Konferenz hatte die Hypertext-Community den Eindruck, dass sie in der Wissenschaft schon weiter vorangeschritten war. Das Konzept von TBL sollte ein „Rückschritt gewesen sein“.

Alan Kay nennt als Gegenargument das fehlen der mathematischen Rigorosität, darüber hinaus sollte es bei TBLs Proposal möglich sein, dass „broken links“ (=Hyperlinks, die zu nichts führen) existieren könnten. Das WWW würde verteilte Texte involvieren, Kay würde lieber verteilte Objekte sehen – generell begründet er die Objekt-Orientierung mit den Aussagen:

„So etwas wie HTML ist genau das, was wir verhindern wollten.“

„Texte ohne klar definierte Semantik sind die Ursache allen Übels.“

„Die Geschichte des WWW ist eine lange Liste von Patches.“ (siehe Vorlesung)

Was sind die fachlichen Argumente, warum das WWW dennoch ein Erfolg wurde?

Zum Erfolg des World Wide Web hat vor allem dazu beigetragen, dass weder TBL noch das CERN Institut sich um Patente bemüht und ihre Erfindungen dadurch geschützt haben. Das hat Entwickler und Unternehmen dazu motiviert, Browser, Webserver-Software und Webseiten-Inhalte zu entwickeln.

Um zu verhindern, dass eine Vielzahl von Webstandards entstehen, gründete TBL 1994 am MIT das W3C (World Wide Web Consortium). Dieses kümmert sich um die Standardisierung der Techniken im WWW, wie z.B. HTML, XHTML, XML, CSS, usw.

Was wäre der Preis für die garantierte Verhinderung von “broken links”?

Der Preis bei der Referentiellen Integrität liegt vor allem im Bereich der Rechnerressourcen, insbesondere der Zeit. Bei der Überprüfung einer relationalen Datenbank wird also Zeit beansprucht. Weiterhin wird eine zentrale Kontrolle benötigt, um broken Links überhaupt verhindern zu können.

1.2. HTTP

Sie bekommen im Browser den HTTP Status Code 200. Was bedeutet das?

Status Code 200 → OK

Hierbei handelt es sich um einen „Success“-Statuscode.

Dies bedeutet, dass bei der Anfrage alles erfolgreich bearbeitet wurde und dies bei der Antwort des HTTP Daemons übertragen wurde.

Sie bekommen im Browser den HTTP Status Code 301. Was hat das zu bedeuten?

Status Code 301 → Moved Permanently

Die angefragte Ressource ist unter der vorhandenen Adresse nicht mehr zu erreichen, die alte Adresse ist nicht länger gültig.

Es erfolgt ein Redirect über das „Location“-Feld im Header, was zur neuen Adresse führt.

Sie bekommen im Browser den HTTP Status Code 400. Was hat das zu bedeuten? Was können Sie dagegen tun?

Status Code 400 → Bad Request

Wird dann geworfen, wenn der Server die Anfrage nicht bearbeiten konnte. Dies basiert allerdings auf einem client-seitigen Fehler, zum Beispiel, wenn eine fehlerhafte Syntax angegeben wurde.
Beispiel: Webadresse falsch eingegeben.

In jedem Fall muss auf der Client-Seite der Fehler behoben werden, bevor mit dem Prozess fortgefahren werden kann. Wurde beispielsweise die URL falsch eingegeben, muss diese zuvor korrigiert werden.

Sie bekommen im Browser den HTTP Status Code 403. Was hat das zu bedeuten? Was können Sie dagegen tun?

Status Code 403 → Forbidden

Dieser Code wird geworfen, sollte der Client nicht die nötigen Rechte haben, um auf eine Adresse zuzugreifen. Der Server hat die Anfrage verstanden, weigert sich aber, sie zuzulassen.

Kann auch passieren, wenn eine als HTTPS konfigurierte seite mit HTTP aufgerufen wurde.

Der Zustand ist dauerhaft und kann nur von einem entsprechenden Netzadministrator beseitigt werden. Die Rechte eines Nutzers müssten geändert werden, um auf die gewünschte Ressource zugreifen zu können.

In einer Webanwendung benötigen Sie eine OPTIONS-Anfrage, die die Optionen des Servers vor dem eigentlichen Zugriff erfragen soll. Aus Performanzgründen soll die Abfrage jedoch cacheable sein. Wie könnten Sie dafür eine Lösung angehen?

OPTIONS-Abfragen sind nicht cacheable.

1.3. HTML-Literatur lesen und Fragen beantworten

Was ist HTML?

Die Abkürzung HTML steht für Hyper Text Markup Language. HTML ist eine sogenannte Auszeichnungs-Sprache und dient dem textbasierten Strukturieren von Dokumenten. Sie wird vor allem für das Erstellen von Webseiten genutzt. HTML ist kontextfrei. Verschiedene Tags werden genutzt, um einem Browser klar zu machen, wie er den Content darstellen muss. Mit diesen kann Text strukturiert, Bilder dargestellt, Anchor-Links eingefügt oder Elemente formatiert werden. Tags werden immer in „< >“ Klammern verpackt und müssen stets mit „</>“ beendet werden.

Beispiel: <p>Test</p>

HTML Code wird gewöhnlicherweise in einer Datei gespeichert, die von Browsern gelesen und grafisch dargestellt werden können.

HTML-Dokumente werden stets mit <!DOCTYPE html> eingeleitet. HTML-Dokumente sind typischerweise in <head> und <body> unterteilt. Im Head können Titel und Metadaten stehen, der Body ist das, was im Browser tatsächlich zu sehen ist.

Aufgrund der Tatsache, dass HTML sehr simpel und intuitiv aufgebaut ist, ist HTML sehr beliebt.

Wie kann man eine geschachtelte geordnete Liste der Schachtelungstiefe 3 erzeugen?

Generell: Für geordnete Listen wird das Tag <ol> genutzt. Um Listeneinträge zu machen, nutzen wir das <li>-Tag.

Für eine Liste der Schachtelungstiefe 3 müssen wir darauf achten, jeweils an der gewünschten Stelle eine neue geordnete Liste mit einem weiteren <ol>-Tag einzuleiten. Wir verschachteln also mehrere <ol>-Tags ineinander, bis wir an der gewünschten Tiefe angelangt sind.

Wie ist eine HTML-Tabelle aufgebaut?

Jede HTML-Tabelle ist ein Zeilen (waagerecht) und Spalten (senkrecht) gegliedert. Der Aufbau wird mit dem <table>-Tag gestartet. Dieses leitet eine Tabelle ein.

Eine neue Zeile in der Tabelle kann mit dem Tag <tr> (table row) eingefügt werden. Innerhalb dieses Tags werden weitere <td> (table data) Elemente genutzt, um die Tabelle mit Werten zu füllen.

Optional kann die erste Zeile mit <th> (table header) Elementen gefüllt werden. Diese dienen dann als einfacher Header einer Tabelle.

Welche Konventionen sollte man bei Pfaden und Dateinamen beachten?

HTML-Dateien enden stets auf .html oder .htm.

HTML-Dateien mit Server Side Includes erhalten die Endung .shtml oder .shtm.

PHP Programmcode Dateien erhalten die Endung .php.

Default-Dateien werden „index.html“/“index.htm“ benannt.

Prinzip: „Cool URIs don't change!“ → URIs so anlegen, dass sie niemals geändert werden müssen und stets aktuell bleiben und die Bedeutung dieser aus dem Namen bereits herausgelesen werden kann (Eindeutigkeit).

Unterverzeichnisse sollten sinnvoll benannt werden. Bilddateien sollten beispielsweise unter /img/ hinterlegt werden, Script-Dateien unter /script/.

Am Einfachsten ist es, dem ISO-Standard zur Identifizierung von Ressourcen aller Art in einem weltweiten „universellen Informationsraum“ zu folgen.

Wie baut man in HTML ein Menü?

Je nachdem, was für eine Art von Menü man realisieren möchte, gibt es verschiedene Methoden, ein Menü zu erstellen.

Ein einfaches Dropdown-Menü lässt sich mit der Hilfe des <select>-Tags in Kombination mit mehreren verschachtelten <option>-Tags erstellen.

Select wird dabei genutzt, um das Dropdown-Menü einzuleiten, die Option Tags dienen dann als Auswahlmöglichkeiten des Menüs.

Eine weitere Möglichkeit wäre es, verschachtelte Listen zu nutzen. Wir eröffnen mit einem <nav>-Element, um ein Menü als Navigationsleiste einzuleiten und verschachteln daraufhin <ul> und <li>-Elemente, um das Menü aufzubauen. Dieses Menü würde wie eine Art „Bullet-Point“-Liste aussehen. Unterpunkte in dieser Liste könnten mit weiteren Verschachtelungs-Stufen, die durch <ul>-Tags eingeleitet werden, erreicht werden.

Zuletzt gibt es auch das eigentliche <menu>-Tag, was eine Liste/ein Menü von Befehlen definiert und ebenfalls für Kontextmenüs, Symbolleisten, etc. genutzt wird. Die Liste wird mit <menu> eingeleitet, gefolgt von einer Reihe von <li>- oder <menuitem>-Elementen.

Welche Attribute sollte man bei Bildern wie verwenden?

<img>-Tags haben zwei notwendige Attribute:

src: referenziert den Pfad zum Bild

alt: Textdisplay, wenn das Bild nicht geladen werden kann

<width> und <height> können sich ebenfalls als nützlich erweisen, wenn man das Bild unter Umständen anderweitig skalieren möchte.
